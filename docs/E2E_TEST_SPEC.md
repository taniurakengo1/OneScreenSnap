## 1. はじめに

### 1.1 目的

本ドキュメントは、macOS向けメニューバーアプリ「OneScreenSnap」のE2E（End-to-End）テスト仕様を定義する。SHIFT品質テスト手法に基づき、100人のプロペルソナ視点でテストケースを網羅的に設計し、アプリケーションの品質を多角的に検証する。

### 1.2 対象アプリ概要

| 項目 | 内容 |
|------|------|
| アプリ名 | OneScreenSnap |
| バージョン | 1.0.0 |
| プラットフォーム | macOS 14.0 (Sonoma) 以上 |
| 種別 | メニューバー常駐アプリ（.app バンドル） |
| 言語 | Swift / AppKit |
| 主要機能 | マルチディスプレイ環境での指定ディスプレイキャプチャ |
| フレームワーク | ScreenCaptureKit, CoreGraphics, AppKit |

### 1.3 テスト環境要件

| 要件 | 最小 | 推奨 |
|------|------|------|
| macOS | 14.0 (Sonoma) | 15.x (Sequoia) |
| ディスプレイ数 | 1台 | 2台以上（外部ディスプレイ含む） |
| Node.js | 18.x | 20.x |
| Playwright | 1.40+ | 最新安定版 |
| Xcode CLI Tools | インストール済み | 最新版 |
| cliclick | インストール済み | 最新版 |
| ffmpeg | インストール済み | 最新版 |
| 権限 | アクセシビリティ、画面収録 | 同左 |

| 用語 | 定義 |
|------|------|
| **ペルソナ** | 特定の専門性・視点を持つ仮想テスター。テストケース設計の起点 |

## 2. テスト方針

### 2.1 SHIFT品質テスト方法論

SHIFT品質テスト手法の核心は、「テスト対象を多角的な専門家視点で検証する」ことにある。単一のテスターでは見落としがちな品質問題を、100人のプロペルソナを通じて体系的に発見する。

**基本原則:**
- **網羅性**: 機能テストだけでなく、権限・パフォーマンス・ローカライゼーション・エッジケースまで包括的にカバー
- **ペルソナ駆動**: 各テストケースは1人以上のペルソナの視点に紐づく
- **サイクルベース**: テストは「サイクル」単位で実行し、1件でもFAILがあれば修正後に全サイクルを再実行
- **エビデンス必須**: 全テストケースに対してスクリーンショット・動画・ログ等のエビデンスを収集

### 2.2 100ペルソナアプローチ

100人のペルソナを12カテゴリに分類し、各ペルソナが「自分ならどうテストするか」を具体的なテストケースとして反映する。

| カテゴリ | 人数 | フォーカス |
|----------|------|-----------|
| A: 一般ユーザー | 10名 | 基本操作フロー、初回起動体験 |
| B: マルチディスプレイ専門 | 10名 | ディスプレイ検出、配置、ホットプラグ |
| C: ショートカット・入力専門 | 10名 | キーバインド、修飾キー、競合 |
| D: キャプチャ機能専門 | 10名 | 全画面/矩形/複数キャプチャ品質 |
| E: リージョンプリセット専門 | 10名 | 範囲選択、保存、復元、座標系 |
| F: 設定画面UI専門 | 10名 | レイアウト、操作性、状態管理 |
| G: 権限・セキュリティ専門 | 8名 | macOS権限、サンドボックス |
| H: パフォーマンス・安定性専門 | 8名 | メモリ、レイテンシ、長時間稼働 |
| I: ローカライゼーション専門 | 8名 | 日英切替、文字列、レイアウト |
| J: エッジケース・異常系専門 | 8名 | 境界値、異常入力、リカバリ |
| K: フィードバック・通知専門 | 4名 | 音、フラッシュ、エラー通知 |
| L: インストール・ライフサイクル専門 | 4名 | インストール、アップデート、起動 |

### 2.3 サイクルベーステストルール

```
テストサイクル実行
  ├─ 全テストPASS → サイクル合格 → 完了
  └─ 1件以上FAIL → バグ修正 → 全サイクル再実行（合格済みも含む）
       └─ エビデンス再取得必須
```

### 2.4 エビデンス収集方針

- 全テストケースに対してスクリーンショットまたは動画を収集
- クリップボード状態の検証にはSwiftブリッジツールを使用
- UserDefaultsの状態確認にはdefaults readコマンドを使用
- テスト実行のPlaywright HTMLレポートを生成

### 3.3 Swiftブリッジツール

テスト専用のSwiftコマンドラインツールを用意し、Playwrightからexecで呼び出す。

#### ax-inspector

Accessibility APIを使ってUIエレメントを検査する。

```
用途:
  - メニューバーアイテムの存在確認
  - 設定画面のUI要素の状態取得
  - ウィンドウの位置・サイズ取得

出力形式: JSON
  {
    "role": "AXMenuBarItem",
    "title": "OneScreenSnap",
    "position": { "x": 800, "y": 0 },
    "size": { "width": 22, "height": 22 },
    "children": [...]
  }
```

#### clipboard-info

クリップボードの内容を検査する。

```
用途:
  - クリップボード内の画像サイズ取得
  - 画像フォーマット確認
  - ピクセル値の検証

出力形式: JSON
  {
    "hasImage": true,
    "width": 1920,
    "height": 1080,
    "format": "PNG",
    "bytesPerRow": 7680
  }
```

## 6. テストサイクル実行ルール

### 6.1 サイクル概要

テストサイクルとは、全テストスイート（TS-INSTALL〜TS-LIFECYCLE）を一通り実行する単位である。

```
サイクル実行フロー:

  [サイクルN開始]
       │
       ▼
  全テストスイートを順次実行
  (TS-INSTALL → TS-LAUNCH → ... → TS-LIFECYCLE)
       │
       ▼
  ┌─ 全テストPASS？ ─┐
  │                    │
  Yes                  No
  │                    │
  ▼                    ▼
  サイクルN合格      バグ報告 & 修正
  (完了)              │
                      ▼
                 [サイクルN+1開始]
                 (全テスト再実行)
                 (エビデンス再取得)
```

### 6.2 サイクルID管理

| 項目 | 形式 | 例 |
|------|------|----|
| サイクルID | `CYCLE-{通番}` | CYCLE-001 |
| 開始日時 | ISO 8601 | 2026-03-19T10:00:00+09:00 |
| 終了日時 | ISO 8601 | 2026-03-19T18:30:00+09:00 |
| ステータス | PASS / FAIL | FAIL |
| FAIL件数 | 整数 | 3 |
| 備考 | 自由記述 | TC-CAP-005で画像サイズ不一致 |

### 6.3 テストケース実行記録

各テストケースの実行結果を以下の形式で記録する:

| 項目 | 内容 |
|------|------|
| サイクルID | CYCLE-001 |
| テストケースID | TC-CAP-001 |
| 実行日時 | 2026-03-19T10:15:00+09:00 |
| 結果 | PASS / FAIL / SKIP / BLOCKED |
| エビデンスファイル | `evidence/CYCLE-001/TC-CAP-001/` |
| 備考 | 問題なし / バグ番号参照 |

### 6.4 合格判定基準

- **サイクル合格**: 全テストケースがPASSまたはSKIP（正当な理由付き）
- **SKIP条件**: ハードウェア非対応（例: トリプルディスプレイテストで2台しかない場合）
- **BLOCKED**: 前提条件のテストがFAILのため実行不能
- **サイクルFAIL → 全再実行**: 1件でもFAILがあれば修正後に新サイクルで全テスト再実行

### 6.5 再実行ルール

1. **FAILしたテストの修正**: バグチケットを起票し、修正を実装
2. **全テスト再実行**: 修正による副作用を検出するため、PASSしたテストも含め全て再実行
3. **エビデンス再取得**: 新サイクルでは全テストケースのエビデンスを新規取得
4. **前サイクルのエビデンスは保持**: 比較・回帰分析のため削除しない

### 8.4 既知の制限

| # | 制限事項 | 影響 | 備考 |
|---|---------|------|------|
| 1 | PlaywrightはmacOSネイティブUIを直接操作できない | テストはosascript/cliclick/Accessibility APIで代替 | ハイブリッド構成で対応 |
| 2 | CGEventTapはアクセシビリティ権限が必要 | テスト環境に事前権限付与が必要 | 権限テスト自体は手動操作が一部必要 |
| 3 | ScreenCaptureKit権限は初回プロンプト | テスト自動化時に手動承認が必要な場合あり | tccutil resetで初期化可能 |
| 4 | ミラーリングモードのテスト | 実機でのみ検証可能 | CI環境では困難 |
| 5 | 縦置き（回転）ディスプレイのテスト | 物理的な回転が必要 | ソフトウェア回転で代替可能 |
| 6 | 複数矩形キャプチャUI | インフラ（ImageComposer）のみ実装済み、UI未実装 | TC-RCAP-009はAPI直接呼び出しでテスト |
| 7 | ファイル保存機能 | 未実装 | テスト対象外 |
| 8 | 履歴機能 | 未実装 | テスト対象外 |
| 9 | UserDefaultsは`make uninstall`で削除されない | macOSの仕様 | `defaults delete`で手動削除可能 |
| 10 | ディスプレイホットプラグのテスト自動化 | 物理操作が必要 | USB-Cドック + cliclickで一部自動化可能 |

### 8.5 テストデータ

テストで使用する固定データ:

```
テストショートカット:
  - ディスプレイ1: ⌘⇧F1 (keyCode: 0x7A, modifiers: Command|Shift)
  - ディスプレイ2: ⌘⇧F2 (keyCode: 0x78, modifiers: Command|Shift)
  - リージョン1:   ⌘⇧F3 (keyCode: 0x63, modifiers: Command|Shift)

テストリージョン:
  - 100×100 at (0, 0)    — 左上
  - 400×300 at (100, 100) — 中央付近
  - 10×10 at (500, 500)   — 最小サイズ
  - 全画面               — ディスプレイ全体

UserDefaultsキー:
  - displayShortcutBindings: ディスプレイショートカット設定
  - regionPresets: リージョンプリセット設定
  - feedbackMode: フィードバックモード (0/1/2)
  - captureResizeMode: リサイズモード (0/1)
```
